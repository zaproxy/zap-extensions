Правила пассивного сканирования - бета

В это дополнение включены следующие правила пассивного сканирования бета-версии:

Обнаружено большое перенаправление (потенциальная утечка конфиденциальной информации)

Эта проверка прогнозирует размер ответов различных типов перенаправления и создает предупреждение, если ответ превышает прогнозируемый размер. Большой ответ о перенаправлении может указывать на то, что, несмотря на то, что перенаправление имело место, страница на самом деле содержала контент (который может раскрывать конфиденциальную информацию, персональные данные и т. д.).

Последний код: BigRedirectsScanRule.java

Отравление файлами cookie

Эта проверка просматривает введенные пользователем данные в параметрах строки запроса и данные POST, чтобы определить, где могут контролироваться параметры файлов cookie. Это называется атакой с отравлением файлов cookie, и ее можно использовать, когда злоумышленник может манипулировать файлами cookie различными способами. В некоторых случаях это невозможно использовать, однако разрешение параметрам URL задавать значения файлов cookie обычно считается ошибкой.

Последний код: UserControlledCookieScanRule.java

Заголовок Content Security Policy (CSP) не задан

Это проверяет заголовки ответа HTML на наличие заголовка Content Security Policy.
По умолчанию это правило проверяет наличие заголовка Content-Security-Policy, а при низком пороге также проверяется наличие заголовков «X-Content-Security-Policy» и «X-WebKit-CSP».
Перенаправления и ответы, отличные от HTML, игнорируются, за исключением нижнего порога.

Если в ответе обнаружен заголовок «Content-Security-Policy-Report-Only», возникает предупреждение INFO. Это может представлять собой правоприменение которая активно совершенствуется или разрабатывается, или реализуется лишь частично.

Последний код: ContentSecurityPolicyMissingScanRule.java

Как правило, это довольно точный способ определить, где перенаправления 301 или 302 могут быть использованы спамерами или фишинговыми атаками.

Раскрытие персональных данных

PII — это такая информация, как номер кредитной карты, SSN и т. д. Эта проверка в настоящее время сообщает только числа, которые соответствуют номерам кредитных карт и передают контрольную сумму Луна, что дает высокую уверенность в том, что это номер кредитной карты (изображения и CSS игнорируются).
При пороге СРЕДНИЙ и ВЫСОКИЙ он пытается использовать три символа контекста с каждой стороны потенциальных совпадений, чтобы исключить совпадения внутри содержимого, похожего на десятичное число. При НИЗКОМ пороге для таких совпадений будут выдаваться оповещения.

Примечание. В случае подозрения на стоимость кредитной карты потенциальные номера кредитных карт проверяются в списке банковских идентификационных номеров. (БИНСписок). Если совпадение найдено, выдается предупреждение с высокой достоверностью, а дополнительные сведения добавляются в поле «Другая информация» в alert, в противном случае оповещения будут иметь средний уровень достоверности. Дополнительную информацию см. в разделе binlist-data.

Последний код: PiiScanRule.java

Обратный переход

При этом проверяется, используют ли какие-либо ссылки целевой атрибут с использованием ключевого слова opener в атрибуте rel, поскольку это может позволить целевым страницам занять место страницы, которая их открывает.
По умолчанию это правило будет игнорировать все ссылки, находящиеся в том же контексте, что и страница. При НИЗКОМ пороге будут игнорироваться только ссылки, находящиеся на одном хосте.
При ВЫСОКОМ пороге он будет сообщать только о ссылках, которые используют цель "_blank".
Вы можете указать список шаблонов регулярных выражений URL, разделенных запятыми, используя параметр rules.domains.trusted на панели «Параметры» «Конфигурация правила». Любой URL-адрес ссылки, соответствующий одному из этих шаблонов, будет считаться находящимся в доверенном домене и, следовательно, о нем не будет сообщено.

Строгий заголовок безопасности транспорта

Это правило проверяет ответы HTTPS на наличие заголовка HTTP Strict Transport Security (HSTS) и тестирует различные проблемы реализации, оповещение, если они будут найдены. Сгенерированные оповещения:
  • Strict-Transport-Security Header Not Set: если ответ HTTPS и заголовок полностью отсутствует.
  • Strict-Transport-Security Disabled: если ответом является HTTPS, а значение max-age установлено равным нулю, таким образом сбрасывается информация HSTS браузера для сайта.
  • Strict-Transport-Security Multiple Header Entries (не соответствует спецификации): если ответ HTTPS и в ответе есть более одного заголовка HSTS.
  • Отсутствует параметр Strict-Transport-Security Max-Age (не соответствует спецификации): если ответ HTTPS, указывается заголовок HSTS, но значение max-age не содержит числа.
  • Strict-Transport-Security определено через META (не соответствует спецификации): если тело ответа включает тег META, который пытается определить HSTS.
  • Заголовок Strict-Transport-Security в простом ответе HTTP: если ответ является HTTP и присутствует заголовок HSTS.
  • Strict-Transport-Security Max-Age искажен (не соответствует спецификации): если ответ HTTPS и заголовок HSTS присутствует, но есть кавычки, предшествующие директиве max-age (кавычки разрешены вокруг значения max-age, но не вокруг самой директивы).
  • Strict-Transport-Security Искаженный контент (не соответствующий спецификации): если ответ HTTPS и заголовок HSTS присутствует, но есть непредвиденный контент (например, фигурные кавычки). Ожидается, что содержимое будет печататься в формате ASCII.
О перенаправлениях на URL-адреса HTTPS в том же домене будет сообщаться только при низком пороге.

Последний код: StrictTransportSecurityScanRule.java
