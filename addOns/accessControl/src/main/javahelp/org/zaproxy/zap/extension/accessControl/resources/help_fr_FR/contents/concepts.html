Tests de Contrôle d'Accès

Ce module permet aux utilisateurs de comparer les différentes parties d’une application web accessibles selon les profils, d’effectuer des tests de contrôle d’accès et d’identifier d’éventuelles failles de sécurité liées aux accès. Il permet la configuration de règles d'accès et de mener d'attaques complètes dans le but d'identifier les sections d'une application web qui seraient accessibles à des utilisateurs non autorisés.

Il existe deux concepts principaux liés à ce module : les règles d'accès et la procédure de test.

Règles d'accès

Afin d’identifier d’éventuels problèmes de contrôle d’accès, ZAP doit savoir quelles parties de l’application web sont censées être accessibles à quels utilisateurs. Dans ZAP, ces règles sont appelées : Règles d’accès et signifient généralement : « PageA devrait / ne devrait pas être accessible à l’utilisateur UserX ».

Les règles sont configurées pour un contexte et, pour chaque utilisateur de ce contexte, chaque nœud du site (page web) est associé à l’une des valeurs suivantes :

  • Autorisé - la ressource est accessible à l’utilisateur auquel la règle s’applique
  • Refusé - la ressource n’est pas accessible à l’utilisateur auquel la règle s’applique
  • Inconnu– il n’y a aucune information quant au fait que la ressource doive ou non être accessible à l’utilisateur auquel la règle s’applique

Afin de simplifier le processus de définition des règles d’accès, ZAP utilise une structure arborescente des URL. Lors de l’analyse des règles, un algorithme d’inférence est utilisé pour déterminer les règles correspondantes pour chaque nœud, en se basant sur son parent dans l’URL, lorsqu’aucune règle spécifique n’est définie Cela signifie que, lors de la configuration des règles d’accès, une seule règle doit être définie explicitement pour un sous-arbre entier, tandis que pour les autres nœuds, les règles sont inférées. Plus de détails à ce sujet sont disponibles sur la page d’aide Options de contextedu contrôle d’accès.

Procédure de Test

Dans l’ensemble, afin de réaliser pleinement des tests de contrôle d’accès pour une application web, les étapes suivantes doivent être suivies :

  • Le testeur définit un ensemble d’utilisateurs et la manière dont ils s’authentifient.
  • Le testeur définit comment ZAP peut identifier les requêtes non autorisées (via le panneau Autorisation des propriétés de session).
  • L’application web est explorée soit manuellement, soit via le Spider ;
  • Le testeur définit les règles d’accès pour chacun des utilisateurs associés au contexte, afin d’indiquer à ZAP quelles parties de l’application web doivent être accessibles à quels utilisateurs.
  • Une 'attaque' est effectuée par ZAP en tentant d’accéder à chaque URL de l’application web du point de vue de chaque utilisateur ;
  • Dans l’onglet Status correspondant, les résultats sont affichés, indiquant quelles pages ont été accessibles avec succès par quels utilisateurs et signalant les cas où les règles d’accès n’ont pas été respectées.
Remarque : Le test de contrôle d’accès n’est pas autorisé en mode Mode sécurisé ni en mode Protected si le contexte n’est pas dans le périmètre.

Alertes

Les alertes suivantes peuvent être générées par l’extension :

Problème de contrôle d’accès – Authentification incorrecte

Une authentification insuffisante se produit lorsqu’un site web permet à un attaquant d’accéder à du contenu ou à des fonctionnalités sensibles sans s’authentifier correctement. Les outils d’administration basés sur le web constituent un bon exemple de sites web offrant un accès à des fonctionnalités sensibles. Selon la ressource en ligne concernée, ces applications web ne devraient pas être directement accessibles sans que l’utilisateur ait correctement vérifié son identité.

Dernier code : AccessControlScannerThread.java
Identifiant de l’alerte : 10101.

Problème de contrôle d’accès – Autorisation insuffisante

Une autorisation insuffisante survient lorsqu’une application n’effectue pas de contrôles d’autorisation adéquats afin de garantir que l’utilisateur exécute une fonction ou accède à des données conformément à la politique de sécurité. Les procédures d’autorisation doivent définir ce qu’un utilisateur, un service ou une application est autorisé à faire. Lorsqu’un utilisateur est authentifié sur un site web, cela ne signifie pas nécessairement qu’il doit avoir un accès complet à l’ensemble du contenu et des fonctionnalités.

Dernier code : AccessControlScannerThread.java
Identifiant de l’alerte : 10102.

API

L’extension expose les points de terminaison d’API suivants :

Actions

analyse

Démarre une analyse du contrôle d’accès avec l’identifiant de contexte et l’identifiant utilisateur donnés (peut être une liste d’identifiants séparés par des virgules). (Paramètres optionnels : booléen indiquant si un utilisateur non authentifié doit être inclus (false par défaut), booléen indiquant si des alertes doivent être générées (true par défaut), et niveau de risque des alertes (High par défaut).)Remarque :Cela suppose que les règles de contrôle d’accès ont été préalablement définies via l’interface graphique de ZAP et que le contexte nécessaire a été exporté/importé.

writeHTMLreport

Génère un rapport de contrôle d’accès pour l’identifiant de contexte donné et l’enregistre en utilisant le nom de fichier (chemin) fourni.

Vues

getScanProgress

Retourne la progression du scan de contrôle d’accès (pourcentage entier) pour l’identifiant de contexte donné.

getScanStatus

Retourne le statut du scan du Contrôle d'Accès (chaîne de caractères) pour l’identifiant de contexte donné.
