Tests de Contrôle d'Accès

Ce module permet aux utilisateurs de comparer les différentes parties d’une application web accessibles selon les profils, d’effectuer des tests de contrôle d’accès et d’identifier d’éventuelles failles de sécurité liées aux accès. Il permet la configuration de règles d'accès et de mener d'attaques complètes dans le but d'identifier les sections d'une application web qui seraient accessibles à des utilisateurs non autorisés.

Il existe deux concepts principaux liés à ce module : les règles d'accès et la procédure de test.

Règles d'accès

Afin d’identifier d’éventuels problèmes de contrôle d’accès, ZAP doit savoir quelles parties de l’application web sont censées être accessibles à quels utilisateurs. Dans ZAP, ces règles sont appelées : Règles d’accès et signifient généralement : « PageA devrait / ne devrait pas être accessible à l’utilisateur UserX ».

Les règles sont configurées pour un contexte et, pour chaque utilisateur de ce contexte, chaque nœud du site (page web) est associé à l’une des valeurs suivantes :

  • Autorisé - la ressource est accessible à l’utilisateur auquel la règle s’applique
  • Refusé - la ressource n’est pas accessible à l’utilisateur auquel la règle s’applique
  • Inconnu– il n’y a aucune information quant au fait que la ressource doive ou non être accessible à l’utilisateur auquel la règle s’applique

Afin de simplifier le processus de définition des règles d’accès, ZAP utilise une structure arborescente des URL. When analyzing the rules, an inference algorithm is used to detect the matching rules for each node based on its parent in the URL if no particular rules are defined. This means that, when configuring the access rules, only 1 rule needs to be set explicitly for an entire subtree, while for the other nodes rules are inferred. More details about this can be found on the Access Control Context options help page.

Procédure de Test

As a whole, in order to fully perform access control testing for a web application, the next steps should be followed:

  • the tester defines set of Users and how they authenticate;
  • the tester defines how ZAP can identify un-authorized requests (via the Authorization panel in the Session Properties);
  • the web-application is explored either manually or via the Spider;
  • the tester define the Access Rules for each of the Users associated to the Context, basically letting ZAP know which parts of the web application are supposed to be accessed by which users;
  • an ‘attack’ is performed by ZAP by trying to access every URL of the web-app from the perspective of every user;
  • in the corresponding Status Tab, the results are displayed, showing which pages were accessed successfully by which users and marking the cases where the access rules where not followed.
Remarque : Le test de contrôle d’accès n’est pas autorisé en mode Safe ni en mode Protected si le contexte n’est pas dans le périmètre.

Alertes

The following Alerts may be raised by the add-on:

Access Control Issue - Improper Authentication

Insufficient Authentication occurs when a web site permits an attacker to access sensitive content or functionality without having to properly authenticate. Web-based administration tools are a good example of web sites providing access to sensitive functionality. Depending on the specific online resource, these web applications should not be directly accessible without requiring the user to properly verify their identity.

Latest Code: AccessControlScannerThread.java
Alert ID: 10101.

Access Control Issue - Improper Authorization

Insufficient Authorization results when an application does not perform adequate authorization checks to ensure that the user is performing a function or accessing data in a manner consistent with the security policy. Authorization procedures should enforce what a user, service or application is permitted to do. When a user is authenticated to a web site, it does not necessarily mean that the user should have full access to all content and functionality.

Latest Code: AccessControlScannerThread.java
Alert ID: 10102.

API

The add-on exposes the following API endpoints:

Actions

scan

Starts an Access Control scan with the given context ID and user ID (can be comma separated list of IDs). (Optional parameters: boolean identifying if an unauthenticated user should be included (default false), boolean identifying whether or not Alerts are raised (default true), and the Risk level for the Alerts (default High).) [Remarque :This assumes the Access Control rules were previously established via ZAP gui and the necessary Context exported/imported.]

writeHTMLreport

Generates an Access Control report for the given context ID and saves it based on the provided filename (path).

Views

getScanProgress

Retourne la progression du scan de contrôle d’accès (pourcentage entier) pour l’identifiant de contexte donné.

getScanStatus

Retourne le statut du scan du Contrôle d'Accès (chaîne de caractères) pour l’identifiant de contexte donné.
